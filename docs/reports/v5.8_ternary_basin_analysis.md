# v5.8.0 — Ternary Basin Analysis & Transition Metrics

## 1. Motivation

The v5.7.0 ternary topology classifier assigns a final ternary state (+1, 0, -1)
to each BP decoding trajectory. However, it does not characterize *how* the
trajectory arrives at that state: whether it transitions smoothly, oscillates
between basins, or gets trapped near a boundary.

v5.8.0 strengthens the ternary framework with:

- Transition detection metrics (boundary crossings, regime switches)
- Metastability scoring for residual convergence characterization
- Deterministic local basin probing around decoding fixed points

These additions turn the classifier into a robust trajectory topology detector.

## 2. Ternary Topology Recap

The ternary classifier (v5.7.0) maps each iteration of a BP decode to one of
three states:

| State | Meaning |
|-------|---------|
| +1 | Stable success basin (syndrome satisfied, converged) |
| 0 | Boundary / metastable region |
| -1 | Failure basin (syndrome unsatisfied, converged) |

The per-iteration trace and final classification provide a topological view
of the decoding trajectory in observable phase space.

## 3. Transition Detection

v5.8.0 adds four new metrics to the ternary topology output:

**boundary_crossings**: Number of transitions involving state 0. Counts
adjacent pairs (t, t+1) in the ternary trace where one state is 0 and the
other differs. High values indicate the trajectory repeatedly enters and
leaves the boundary region.

**regime_switch_count**: Total number of state changes between any pair of
ternary states (+1, 0, -1). Measures overall trajectory volatility.

**first_success_iteration**: First index in the ternary trace where state
equals +1. Returns null if the trajectory never enters the success basin.

**first_failure_iteration**: First index in the ternary trace where state
equals -1. Returns null if the trajectory never enters the failure basin.

### Example Output

```json
{
  "ternary_trace": [0, 0, 0, -1, -1],
  "final_ternary_state": -1,
  "transition_iteration": 3,
  "boundary_crossings": 1,
  "regime_switch_count": 1,
  "first_success_iteration": null,
  "first_failure_iteration": 3
}
```

## 4. Basin Probing Algorithm

The local basin probe deterministically explores the neighborhood of a
decoding point by perturbing the LLR vector along fixed directions.

### Algorithm

```
INPUT:  llr_vector, decode_function, perturbation_scale, directions
OUTPUT: probe_results with ternary classification per direction

1. Compute baseline_correction = decode(llr_vector)
2. For each direction d_i:
   a. llr_perturbed = llr_vector + perturbation_scale * d_i
   b. perturbed_correction = decode(llr_perturbed)
   c. Classify perturbed_correction into ternary state
3. Compute success/failure/boundary fractions
```

### Direction Selection

Directions are deterministic vectors. Default: standard basis vectors e_i
(unit vectors with a 1 in position i). Custom directions can be provided,
including spectral-mode directions from Tanner graph analysis.

### Classification

When syndrome information is available:
- Syndrome satisfied → +1 (success)
- Syndrome unsatisfied → -1 (failure)

When no syndrome is available:
- Same correction as baseline → +1 (same basin)
- Different correction → 0 (boundary/ambiguous)

### Example Output

```json
{
  "probe_results": [
    {"direction": 0, "state": 1},
    {"direction": 1, "state": 0},
    {"direction": 2, "state": -1}
  ],
  "success_fraction": 0.33,
  "failure_fraction": 0.33,
  "boundary_fraction": 0.33
}
```

## 5. Deterministic Design Constraints

All v5.8.0 additions comply with the architectural constitution:

1. **No randomness**: All perturbation directions are deterministic.
   No RNG state, no Python `hash()`, no stochastic sampling.

2. **Decoder treated as pure function**: The basin probe calls the decoder
   entrypoint exactly as external code would. No hooks, no patches, no
   internal state modification.

3. **No input mutation**: The original LLR vector is never modified in-place.
   All perturbations operate on explicit copies.

4. **Side-effect free**: Diagnostics do not alter decoder behavior, state,
   or execution paths. Removing the diagnostics layer produces identical
   decoding behavior.

5. **JSON serializable**: All outputs use only Python primitives (int, float,
   str, list, dict, None) and survive JSON roundtrip.

6. **Opt-in**: All new features are disabled by default and activated only
   through explicit CLI flags or function parameters.

## 6. Example Topology Maps

### Stable Success Basin

A trajectory that converges smoothly to the success basin:

```
Ternary trace:  [0, 0, 1, 1, 1, 1, 1]
Crossings:      1
Regime switches: 1
Basin probe:     success_fraction=1.0 (homogeneous basin)
Metastability:   0.001 (converged)
```

### Oscillating Trajectory

A trajectory that oscillates between basins:

```
Ternary trace:  [0, 1, 0, -1, 0, 1, 0]
Crossings:      6
Regime switches: 6
Basin probe:     success_fraction=0.4, failure_fraction=0.3, boundary=0.3
Metastability:   0.85 (high oscillation)
```

```
Metastability:   0.02 (low variation, plateau)
```

## 7. Experimental Methodology

### Using Transition Metrics

Enable via CLI:

```bash
python bench/dps_v381_eval.py --ternary-transition-metrics
```

This adds transition metrics and metastability scores to the ternary
topology output for each trial.

### Using Basin Probes

Enable via CLI:

```bash
python bench/dps_v381_eval.py --ternary-basin-probe
```

This performs a deterministic local basin probe around the final LLR state
of each trial. The probe uses standard basis directions with a default
perturbation scale of 1e-3.

### Combined Analysis

```bash
python bench/dps_v381_eval.py \
    --ternary-transition-metrics \
    --ternary-basin-probe
```

Both flags imply `--ternary-topology` and `--bp-phase-space`.

### Interpreting Results

1. **Boundary crossings > 0**: The trajectory crosses between basins.
   Higher values indicate more volatile decoding.

2. **Metastability score < 0.01**: Strong convergence.
   Score 0.01-0.1: Plateau behavior. Score > 0.1: Oscillation.

3. **Basin probe success_fraction near 1.0**: The decoding point is deep
   inside the success basin. Low success_fraction indicates proximity to
   the decision boundary.
